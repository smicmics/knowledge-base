🤖 Claude Desktop & Cowork einrichten

1

Claude Desktop App installieren

Lade die Claude Desktop App von claude.ai/download herunter und führe den Installer aus. Nach der Installation mit deinem Anthropic-Konto anmelden.

  • Windows: .exe-Installer herunterladen und ausführen
  • Nach dem Start: Anmelden mit bestehendem claude.ai-Konto
  • Cowork-Modus ist nach der Anmeldung direkt verfügbar
Hinweis: Cowork ist ein Research Preview — Funktionen können sich noch ändern.
2

Cowork einrichten und Ordnerstruktur anlegen

Beim ersten Start von Cowork wird ein Arbeitsordner auf deinem Computer verknüpft. Lege diesen Ordner vorab im Explorer an, bevor du ihn in Cowork auswählst.

Empfohlene Ordnerstruktur für eine Wissensdatenbank:

# Empfohlene Struktur im Projektordner knowledge-base/ ├── web/ # HTML-Seiten ├── assets/ # Bilder, Icons └── raw-data/ # Quelldateien, CSVs
Achtung: Den Ordner entweder im Explorer anlegen oder über die GitHub-GUI erstellen — nicht beides. Sonst entstehen Konflikte beim ersten Push.

🐧 WSL 2 installieren und Ubuntu einrichten

3

WSL 2 installieren und Ubuntu einrichten

WSL 2 (Windows Subsystem for Linux) ermöglicht die Nutzung von Linux-Tools direkt unter Windows. PowerShell als Administrator öffnen und ausführen:

wsl --install

Nach dem Neustart startet Ubuntu automatisch und fordert zur Anlage eines Linux-Benutzernamens und Passworts auf.

  • Ubuntu-Version prüfen: lsb_release -a
  • Pakete aktualisieren: sudo apt update && sudo apt upgrade -y
  • WSL-Version prüfen: wsl --list --verbose (sollte Version 2 zeigen)
Tipp: Das Linux-Passwort wird beim sudo-Befehl benötigt — sicher notieren.

🔀 Git einrichten und GitHub Account anlegen

4

Git einrichten und GitHub Account anlegen

Git in WSL (Ubuntu) konfigurieren und einen GitHub-Account anlegen, falls noch keiner vorhanden ist.

# Git globale Identität setzen (in WSL/Ubuntu) git config --global user.name "Dein Name" git config --global user.email "deine@email.de" # Konfiguration prüfen git config --list
  • GitHub-Account anlegen unter github.com
  • E-Mail-Adresse bestätigen (Bestätigungsmail kommt automatisch)
  • Benutzername sorgfältig wählen — erscheint in der GitHub Pages URL
Wichtig: Zusätzlich zu WSL Git auch Git für Windows installieren (git-scm.com). VS Code und andere Windows-Tools nutzen die Windows-Version, nicht das WSL-Git.

🔑 Personal Access Token erstellen

5

Personal Access Token (PAT) erstellen

GitHub akzeptiert keine Passwörter mehr für Git-Operationen über die Kommandozeile. Stattdessen wird ein Personal Access Token (PAT) benötigt.

  • GitHub → Profilbild → Settings
  • Linke Sidebar ganz unten: Developer settings
  • Personal access tokens → Tokens (classic)
  • Generate new token (classic) klicken
  • Note: z. B. wsl-push, Ablauf: 90 Tage
  • Scopes: repo vollständig aktivieren
  • Token generieren und sofort kopieren — er wird nur einmal angezeigt!
# Token beim ersten Push als Passwort eingeben Username: dein-github-benutzername Password: ghp_xxxxxxxxxxxxxxxxxxxx # ← Token, nicht das GitHub-Passwort!
Tipp: Token in einem Passwortmanager speichern. Nach dem ersten erfolgreichen Push speichert Git den Token im Credential Store.

📦 Repository anlegen und GitHub Pages aktivieren

6

Repository anlegen und GitHub Pages aktivieren

Ein neues Repository auf GitHub anlegen und GitHub Pages für die automatische Veröffentlichung als Website aktivieren.

  • GitHub → New repository
  • Repository-Name wählen (wird Teil der URL: benutzername.github.io/repo-name)
  • Visibility: Public (für GitHub Pages kostenlos erforderlich)
  • Keinen README, .gitignore oder Lizenz über die GUI hinzufügen, wenn der Ordner lokal bereits existiert

GitHub Pages aktivieren:

  • Repository → Settings → Pages
  • Source: Deploy from a branch
  • Branch: main, Ordner: /root oder /docs
  • Nach dem nächsten Push ist die Seite unter https://benutzername.github.io/repo-name/ erreichbar
Achtung: Den Repository-Namen nicht identisch zum lokalen Ordnernamen wählen, wenn dieser bereits als Remote-URL konfiguriert wurde — das führt zu Verwechslungen beim Klonen.

💻 VS Code installieren mit Extensions

7

VS Code installieren mit Extensions WSL und GitLens

Visual Studio Code (code.visualstudio.com) installieren. Die beiden wichtigsten Extensions für dieses Setup:

# Extensions über die VS Code Kommandozeile installieren code --install-extension ms-vscode-remote.remote-wsl code --install-extension eamodio.gitlens
  • WSL Ermöglicht das Öffnen von WSL-Ordnern direkt in VS Code — Dateisystem, Terminal und Git laufen dabei in Linux
  • GitLens Erweitert die Git-Integration: Blame-Ansichten, Commit-Historie, Branch-Vergleiche direkt im Editor

WSL-Projekt in VS Code öffnen:

# Im Ubuntu-Terminal, im Projektordner: code .
Tipp: VS Code erkennt automatisch, dass der Ordner in WSL liegt, und schlägt vor, die WSL-Extension zu aktivieren. Einmalig bestätigen genügt.

🚀 Ersten Push durchführen

8

Lokales Repository mit GitHub verbinden und pushen

Im WSL-Terminal im Projektordner — Repository initialisieren, Remote setzen und ersten Commit pushen:

# Im Projektordner in Ubuntu/WSL git init git add . git commit -m "feat: initial commit" # Remote hinzufügen (URL von GitHub kopieren) git remote add origin https://github.com/benutzername/repo-name.git # Branch umbenennen (GitHub erwartet 'main') git branch -M main # Ersten Push durchführen (-u setzt Tracking für künftige Pushes) git push -u origin main

Bei der Passwortabfrage den Personal Access Token eingeben (nicht das GitHub-Passwort).

Danach: Weitere Pushes laufen einfach mit git push — Branch und Remote sind durch -u bereits verknüpft.

💡 Lessons Learned

Häufige Stolperfallen beim ersten Einrichten — damit du sie nicht selbst erleben musst.

🏷️

Projektname ≠ Ordnername

Repository-Namen auf GitHub nicht identisch zum lokalen Ordnernamen wählen. Das führt beim Klonen und bei der Remote-Konfiguration zu Verwirrung.

Eindeutige, beschreibende Namen für Ordner und Repository vergeben, die zwar zusammenpassen, aber nicht identisch sind.
🔐

Token statt Passwort bei GitHub

GitHub akzeptiert beim git push kein normales Konto-Passwort mehr. Der Push schlägt mit einem Authentifizierungsfehler fehl.

Personal Access Token (PAT) erstellen und diesen als Passwort eingeben. Token sofort nach der Erstellung kopieren.
📁

Ordner vorab ODER über GUI anlegen

Ordner lokal anlegen und zusätzlich über die GitHub-GUI mit README initialisieren — das erzeugt divergierende Historien, die sich nicht ohne Weiteres zusammenführen lassen.

Entweder den Ordner lokal anlegen und leer auf GitHub pushen — oder GitHub klonen und lokal befüllen. Nicht beides.
🪟

Git für Windows zusätzlich installieren

WSL bringt zwar Git für Linux mit, aber Windows-Tools wie VS Code oder GitLens nutzen die Windows-Git-Instanz. Ohne Git für Windows funktionieren diese nicht korrekt.

Git für Windows von git-scm.com installieren. Beide Git-Instanzen (WSL + Windows) können unabhängig voneinander koexistieren.
🔒

VS Code Push Authentifizierung

VS Code versucht beim ersten Push eine Browser-Authentifizierung bei GitHub — dieser Prozess kann feststecken und den Push blockieren.

Lösung 1: Im VS Code Terminal einmalig ausführen:
git config --global credential.helper manager
Lösung 2: Push immer zuverlässig über das WSL Terminal mit git add, git commit, git push durchführen — das ist für Einsteiger die verlässlichere Methode gegenüber der VS Code Source Control.
