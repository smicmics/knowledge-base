⚔️ Git vs. GitHub — der Unterschied

Die beiden Namen klingen ähnlich und werden oft zusammen genannt, sind aber grundverschiedene Dinge.

🔀 Git

  • Ein Programm, das lokal auf deinem Computer läuft
  • Verwaltet die Versionsgeschichte deiner Dateien
  • Arbeitet vollständig offline
  • Erfunden von Linus Torvalds (2005)
  • Keine Benutzeroberfläche, nur Terminal-Befehle
  • Kostenlos und Open Source

🐙 GitHub

  • Eine Website (github.com) in der Cloud
  • Speichert Git-Repositories online (Remote)
  • Ermöglicht Zusammenarbeit im Team
  • Gehört zu Microsoft (seit 2018)
  • Hat eine grafische Benutzeroberfläche
  • Kostenlos für öffentliche Repositories
Merksatz: Git ist das Werkzeug — GitHub ist die Garage, in der du deine Arbeit aufbewahrst und mit anderen teilst. Git funktioniert auch ohne GitHub, aber GitHub braucht Git.

📖 Die wichtigsten Begriffe

📦

Repository (Repo)

repo

Ein Repository ist der Projektordner, den Git verwaltet. Er enthält alle Dateien des Projekts und die komplette Versionsgeschichte aller jemals gemachten Änderungen. Erkennbar am versteckten Unterordner .git/.

Ein Repository gibt es in zwei Varianten: lokal (auf deinem Computer) und remote (auf GitHub). Beide werden mit git push und git pull synchronisiert.

📸

Commit

git commit

Ein Commit ist ein gespeicherter Schnappschuss des aktuellen Projektzustands. Jeder Commit bekommt eine eindeutige ID (Hash), einen Zeitstempel und eine Beschreibung.

# Dateien für den nächsten Commit vormerken (stagen) git add datei.html git add . # alle geänderten Dateien # Commit erstellen mit Beschreibung git commit -m "feat: Startseite hinzugefügt"

Gute Commit-Messages beschreiben das Warum, nicht das Was. Konvention: typ: kurze beschreibung — z. B. fix:, feat:, docs:.

⬆️

Push

git push

Push überträgt die lokalen Commits zum Remote-Repository auf GitHub. Erst nach einem Push sind die Änderungen online sichtbar — und für andere zugänglich.

# Lokale Commits zu GitHub übertragen git push # Beim ersten Push: Branch und Remote verknüpfen git push -u origin main
⬇️

Pull

git pull

Pull holt Änderungen vom Remote-Repository und integriert sie in den lokalen Stand. Wichtig, wenn mehrere Personen am selben Projekt arbeiten oder Änderungen direkt auf GitHub gemacht wurden.

# Neueste Änderungen von GitHub holen und integrieren git pull # Nur holen, noch nicht integrieren (zum Vergleichen) git fetch
Faustregel: Vor dem Arbeiten immer zuerst git pull — so vermeidet man Konflikte.

🔑 Personal Access Token

GitHub akzeptiert seit 2021 keine normalen Passwörter mehr für git push über HTTPS. Stattdessen wird ein Personal Access Token (PAT) benötigt — ein generiertes Passwort mit eingeschränkten Rechten.

🏭

Token erstellen

GitHub → Profilbild (oben rechts) → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token

Einstellungen für den Token:

Note: wsl-push (frei wählbarer Name) Expiration: 90 days (oder kein Ablauf) Scopes: ☑ repo (alle repo-Berechtigungen)
Einmalig sichtbar: Den Token direkt nach der Erstellung kopieren und sicher speichern. GitHub zeigt ihn nur einmal — danach nie wieder.
🔐

Token verwenden

Beim ersten git push fragt Git nach Benutzername und Passwort. Das Passwort ist der Token — nicht das GitHub-Konto-Passwort.

Username: dein-github-benutzername Password: ghp_xxxxxxxxxxxxxxxxxxxx ← Token hier einfügen

Danach speichert Git die Zugangsdaten automatisch im Credential Store — weitere Pushes verlaufen ohne erneute Eingabe.

VS Code: Beim ersten Push in VS Code öffnet sich ein Browser-Fenster. Dort nicht den Browser-Login nutzen — stattdessen den Reiter „Token" wählen und den PAT einfügen.

🌐 GitHub Pages aktivieren

GitHub Pages verwandelt ein öffentliches Repository in eine statische Website — kostenlos, ohne eigenen Server, ohne Konfiguration. HTML, CSS und JavaScript werden direkt aus dem Repository ausgeliefert.

⚙️

Schritt für Schritt aktivieren

Im GitHub-Repository:

Repository → Settings → Pages Source: Deploy from a branch Branch: main Folder: / (root) ← oder /docs für Unterordner → Save klicken

Nach dem nächsten Push ist die Seite automatisch erreichbar unter:

https://benutzername.github.io/repository-name/
⏱️

Deployment-Zeit und Aktualisierung

Nach jedem git push auf den main-Branch startet GitHub automatisch einen Build-Prozess. Die aktualisierte Website ist nach ca. 1–3 Minuten online.

Den Fortschritt des Deployments verfolgen unter: Repository → Actions — dort wird jeder Build mit Status angezeigt.

Tipp: Die index.html im Root-Verzeichnis (oder im /docs-Ordner) ist die Einstiegsseite. Andere HTML-Dateien sind über ihren relativen Pfad erreichbar.
