🔤 Die Hauptsymbole erklärt

VS Code zeigt im Explorer und im Source Control Panel neben jeder Datei ein Buchstaben-Kürzel. Diese Kürzel spiegeln den Zustand der Datei im Git-Repository wider.

U

U — Untracked (nicht verfolgt)

Die Datei ist neu und Git kennt sie noch nicht. Sie existiert auf dem Dateisystem, ist aber noch nie Teil eines Commits gewesen.

  • Typisch beim Anlegen neuer Dateien oder Ordner
  • Git ignoriert die Datei komplett — kein Backup, keine Versionsgeschichte
  • Mit git add datei.html wird die Datei in die Git-Kontrolle aufgenommen
  • Danach wechselt das Symbol von U zu A (Added)
M

M — Modified (verändert)

Die Datei war bereits in einem früheren Commit enthalten und wurde seitdem verändert. Git hat die Änderungen erkannt, aber noch nicht gespeichert.

  • Erscheint sobald gespeicherte Änderungen vom letzten Commit abweichen
  • Noch kein git add gemacht — Änderungen sind nicht gestaged
  • Ein git add + git commit sichert die neue Version dauerhaft
  • Ein git restore datei.html verwirft alle Änderungen seit dem letzten Commit
A

A — Added / Staged (vorgemerkt)

Die Datei wurde mit git add für den nächsten Commit vorgemerkt (gestaged). Sie wartet darauf, mit git commit dauerhaft gesichert zu werden.

  • Symbol wechselt von U oder M zu A
  • Im Source Control Panel: Datei erscheint unter „Staged Changes"
  • Der Commit wird nur die gestageten Dateien einschließen
  • Mit git restore --staged datei.html kann das Staging rückgängig gemacht werden
✓

Kein Symbol — Committed (gespeichert)

Die Datei ist identisch mit dem letzten Commit. Es gibt keine ungesicherten Änderungen. VS Code zeigt kein Symbol an — das ist der gewünschte Normalzustand nach einem erfolgreichen Push.

📂 Der grüne Punkt am Ordner

VS Code zeigt nicht nur bei einzelnen Dateien, sondern auch bei Ordnern den Git-Status an — als farbiger Punkt in der Explorer-Ansicht.

📁 knowledge-base ← enthält neue/geänderte Dateien
📁 web
📄 index.html U
📄 namenskonventionen.html M

Der farbige Punkt an einem Ordner bedeutet: irgendwo in diesem Ordner gibt es Dateien mit offenem Git-Status. Der Punkt „blubbert" den Status der Kinder nach oben — bis zum Wurzelordner des Repositories.

Farb-Bedeutung: Grün steht für neue ungetrackte Dateien (U), Orange/Gelb für veränderte Dateien (M). Die genaue Farbe hängt vom VS Code Theme ab — der Buchstabe an der Datei ist immer eindeutig.

Symbol Bezeichnung Bedeutung Nächster Schritt

🚀 Was nach dem Push passiert

Ein erfolgreicher Push verändert die Anzeige in VS Code spürbar. Alle Symbole verschwinden — der Arbeitsbereich ist wieder sauber.

⏳ Vor dem Push

  • Dateien zeigen U, M oder A
  • Ordner zeigen farbige Punkte
  • Source Control Badge zeigt Anzahl der Änderungen (z. B. ③)
  • GitHub zeigt noch den alten Stand

✅ Nach dem Push

  • Keine Symbole mehr an Dateien
  • Keine farbigen Punkte an Ordnern
  • Source Control Badge ist leer oder weg
  • GitHub zeigt den aktuellen Stand
  • GitHub Pages baut die Website neu (1–3 Min.)
Ablauf: git add . → alle Symbole wechseln zu A → git commit -m "..." → Symbole verschwinden lokal → git push → GitHub und GitHub Pages sind aktuell.

🤫 Keine Nachricht — gute Nachricht

🟢

Kein Symbol = alles synchron

Wenn VS Code keine Buchstaben-Symbole an Dateien und keine farbigen Punkte an Ordnern anzeigt, ist alles in Ordnung. Der lokale Stand ist identisch mit dem letzten Commit — und nach einem Push auch identisch mit GitHub.

Git ist ein stilles System: Es meldet sich nur, wenn etwas zu tun ist. Silence is success. Wer nach einem Push keine roten Meldungen und keine Symbole mehr sieht, hat alles richtig gemacht.

Status jederzeit prüfen: Im Terminal mit git status — die Ausgabe „nothing to commit, working tree clean" ist die beste aller Möglichkeiten.
